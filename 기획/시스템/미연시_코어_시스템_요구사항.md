# 미연시 코어 시스템 요구사항

## 1. 목적

이 문서는 현재 스토리 기획을 실제 게임 시스템으로 구현하기 위해 필요한 코어 로직을 정리한다.

현재 StoryFlow는 Scene / Shot / Branch / Transition 기반의 스토리 실행 구조를 제공한다. 미연시 게임을 만들기 위해서는 그 위에 날짜, 이벤트 선택, 캐릭터 진행도, 플래그, 루프, 엔딩 기록 같은 게임 상태 관리 계층이 필요하다.

## 2. 현재 StoryFlow 기준

`StoryFlow`의 현재 기능 요약:

- `StorySceneAsset` 하나가 하나의 Scene 흐름을 가짐
- Scene 안에는 Shot / Branch / Transition 그래프가 있음
- Shot은 실제 진행 단계
- Branch는 여러 출력 중 다음 경로 선택
- Transition은 다음 Scene으로 이동
- 저장/복원 최소 단위는 `FStoryFlowRef = SceneID + ShotID`
- `UStoryShotBase`는 `OnEnterShot`, `OnTickShot`, `FinishShot` 흐름을 제공함
- `UStoryBranchBase`는 `SelectNextIndex`에서 다음 출력 인덱스를 고르는 구조임
- `FStorySceneBranchLink`는 Shot, Branch, Scene 이동을 모두 표현함

현재 구조는 장면 실행에는 적합하지만, 미연시의 장기 상태를 직접 표현하기에는 부족하다.

따라서 추가 코어 시스템은 StoryFlow를 대체하지 않고, StoryFlow가 참조할 수 있는 `스토리 상태 레이어`로 설계하는 것이 좋다.

이 상태 레이어와 미연시 전용 Shot/Branch/UI 브리지는 별도 `VisualNovelPlugin`으로 분리한다. StoryFlow는 범용 장면 실행 플러그인으로 유지하고, `VisualNovelPlugin`이 날짜/플래그/선택지/대사창/엔딩 기록 같은 장르 기능을 담당한다. 1차 구현 범위는 [VisualNovelPlugin 1차 구현 범위](./VisualNovelPlugin_1차_구현범위.md)를 정본으로 따른다.

### 2.1 미연시 적용 판단

StoryFlow 자체를 미연시 전용으로 크게 바꾸는 것은 권장하지 않는다.

현재 StoryFlow는 `장면을 실행하고 다음 링크로 이동하는 엔진`으로 두고, 미연시에 필요한 날짜/플래그/호감도/기억 조각/엔딩 기록은 별도 시스템에서 관리하는 편이 안전하다.

필요한 것은 StoryFlow의 대규모 구조 변경이 아니라, 별도 `VisualNovelPlugin`에 두는 미연시 전용 확장 계층이다.

| 필요 영역 | StoryFlow 자체 변경 여부 | 권장 방향 |
| --- | --- | --- |
| 대사 출력, 배경 전환, 음악/SFX | 불필요 | `VisualNovelPlugin`의 `UVNDialogueShot` 추가 |
| 선택지 표시 | StoryFlow 코어 변경은 불필요 | `VisualNovelPlugin`의 `UVNChoiceShot` 또는 선택지 UI 브리지 |
| 조건 분기 | 코어 변경은 불필요 | `VisualNovelPlugin`의 `UVNConditionBranch`에서 `StoryState` 조회 |
| 날짜 허브 | StoryFlow 내부에 넣지 않음 | `VisualNovelPlugin`의 이벤트 허브가 다음 SceneID를 결정 |
| 플래그/호감도/기억 조각 | StoryFlow 내부에 넣지 않음 | `VisualNovelPlugin`의 상태 계층에서 관리하고 SaveGame과 연결 |
| 루프/엔딩 기록 | StoryFlow 내부에 넣지 않음 | `VisualNovelPlugin`의 Loop/Ending 관리 계층에서 처리 |
| 현재 진행 저장 | 기존 `FStoryFlowRef` 활용 가능 | `StoryState` 저장 데이터에 `FStoryFlowRef`를 포함 |

즉, StoryFlow는 `재생기`, 미연시 시스템은 `상태와 일정 관리자`로 역할을 나눈다.

`VisualNovelPlugin`의 1차 최소 후보는 다음 정도이며, 확정 범위와 제외 범위는 [VisualNovelPlugin 1차 구현 범위](./VisualNovelPlugin_1차_구현범위.md)를 따른다.

1. VN용 기본 Shot 템플릿
   - 대사, 캐릭터 표시, 배경, 음악, SFX, 플래그 변경을 처리하는 Shot 파생 클래스.
2. 조건 Branch 템플릿
   - `RequiredFlags`, 점수 조건, 기억 조각 조건을 조회해 다음 링크를 선택.
3. 선택지 Shot 또는 선택지 UI 브리지
   - 플레이어 선택을 받아 상태를 갱신하고 다음 Shot/Scene으로 넘김.
4. 저장 브리지
   - `FStoryFlowRef`와 `StoryState`를 함께 저장/복원.

이 네 가지는 StoryFlow 플러그인의 범용 실행 구조를 유지하면서, 별도 미연시 플러그인으로 재사용 가능한 제작성을 높이는 확장이다.

### 2.2 플러그인 책임 분리

권장 플러그인 구조:

```text
StoryFlowPlugin
- Scene / Shot / Branch 실행
- FStoryFlowRef
- Scene Registry
- Target Level 전환
- 그래프 편집과 디버그 표시

VisualNovelPlugin
- StoryState / 날짜 / 슬롯 / 플래그
- 캐릭터 진행도 / 기억 조각 / 엔딩 기록
- VN용 Dialogue Shot / Choice Shot / Condition Branch
- 이벤트 허브와 루프 처리
- StoryFlowRef + StoryState 저장/복원 브리지

Miyeansi Game Project
- Miyeansi 전용 캐릭터, 루트, 이벤트 데이터
- UI 스킨, 배경/스탠딩 CG, 스토리 에셋
```

분리 원칙:

- StoryFlowPlugin은 미연시 장르 단어를 몰라도 동작해야 한다.
- VisualNovelPlugin은 StoryFlowPlugin에 의존할 수 있지만, StoryFlowPlugin은 VisualNovelPlugin에 의존하지 않는다.
- Miyeansi 전용 이름과 데이터는 VisualNovelPlugin 코드에 하드코딩하지 않는다.
- 나중에 다른 미연시 프로젝트에서도 VisualNovelPlugin을 가져다 쓸 수 있어야 한다.

## 3. 게임 구조 요약

현재 스토리 구조:

```text
첫 회차 D-32 ~ D-26: 기존 히로인 관계 기준점
D-25: 은하연 전학 / 루프 시작점
D-23: 담임의 축제 이틀 일정 공식 공지 / 2학년 5반의 반 단위 참여 확정
D-25 ~ D-19: 하연 적응 / 관계 변화 시작
D-18 ~ D-12: 2학년 5반 반행사 `쉬어가 카페` 확정 / 축제 준비 진입 / 관계 확장
D-11 ~ D-5: 관계 균열 심화 / 사고 단서 구체화
D-4 ~ D-2: 기억 회복 심화 / 관계 정리 클라이맥스 / 하연 고백 예고
D-1 목요일: 축제 1일차 / 부스 정상 운영 / 하연과 `쉬어가 카페` 운영으로 관계 강화
D-Day 금요일: 축제 2일차 / 부스 축소 운영 후 히로인 선택 / 사고 재현 / 엔딩 분기
D+1 토요일: 날짜 카드 없음 / 진엔딩에서만 암전으로 통과
D+2 일요일: 진엔딩 병실 기상 / `HasSeenTrueEndingWake`
D+3 월요일: 방학식 예정 / 실패 루프에서는 도달하지 못하는 다음 월요일 / 진엔딩에서는 `HasReachedTrueEndingMonday` 이후 참석
```
표기 규칙:

- `D-Day`는 내부 날짜 상태, 조건 평가, 엔딩 분기 기준으로만 사용한다.
- 게임 내 대사, 칠판, 일정표, UI에는 `D-Day`를 직접 노출하지 않는다.
- 게임 내 표기는 `축제 1일차/첫날/목요일`, `축제 2일차/마지막 날/금요일`, `폐장 후`를 사용한다.

루프 이후에는 D-32 ~ D-26 구간을 반복하지 않고 D-25 월요일에서 시작한다. D+3 월요일 방학식은 일정상 존재하지만, 실패 루프에서는 D-Day 이후 그 날짜에 도달하지 못한다. 진엔딩에서는 D-Day 금요일 밤 이후 토요일을 날짜 표기로 보여주지 않고 일요일 오후 병실로 점프하며, 재윤이 깨어난 뒤 D+3 월요일 방학식에 겨우 참석한다.

진엔딩 조건은 세 축이다.

1. 하연과의 상호 마음 확인
2. 기존 관계 정리
3. 사고 단서 확보

하연과 서로의 마음을 확인하지 못한 상태로 D-Day 금요일 밤에 도달하면, 재윤은 하연이 있던 사고 기억을 온전히 복구하지 못한다. 이 경우 폐장 후 학교 뒤편 야외 소무대에 혼자 선 사고로 재구성되고, D+3 월요일 방학식에 도달하지 못한 채 D-25 월요일로 루프한다.

## 4. 필수 런타임 상태

### 4.0 네이밍 규칙

현재 코드 스타일을 기준으로 런타임 상태 이름을 정한다.

| 대상 | 규칙 | 예시 |
| --- | --- | --- |
| C++ 멤버 변수 | 기존 코드처럼 앞 `_` + PascalCase | `_CurrentSceneAsset`, `_CanModify` |
| 함수 인자 | 기존 코드처럼 앞 `_` + snake_case | `_scene_id`, `_story_flow_ref` |
| 로컬 변수 | snake_case | `scene_marker_index`, `level_package_name` |
| StoryFlow `FName` ID | `FStorySceneID`, `FStoryShotID`, `FStoryBranchID`에 들어가는 값. 짧은 PascalCase. 필요할 때만 도메인 접두어 사용 | `SF_DDay`, `DDay_05True`, `TE_01_Wake` |
| 저장/조건 키 | 짧은 PascalCase. 문장형 이름 금지 | `HayeonPace`, `IsSohaResolved`, `HasReachedTrueEndingMonday` |
| bool 키 | `b` 접두어 없이 의문문형 PascalCase. 상태는 `Is/Has`, 가능 여부는 `Can`, 의도/옵션은 `Should` 사용 | `IsFirstRun`, `IsDDayTrueEnding`, `CanEnterTrueEndingCredit` |

캐릭터명은 변수/키 안에서 `Hayeon`, `Soha`, `Seorin`, `Miru`로 통일한다. `HaYeon`, `SoHa`, `SeoRin`, `MiRu`처럼 음절마다 대문자를 나누지 않는다.

진엔딩 후일담 전용 상태는 `TE_` 접두어를 사용한다. 예: `HasSeenTrueEndingWake`, `HasReachedTrueEndingMonday`, `CanEnterTrueEndingCredit`.


### 4.0.1 StoryFlow 제작용 시나리오 연결 기준

스토리 구현 기준은 [전체 스토리 제작용 시나리오](../스토리/17_전체_스토리_제작용_시나리오.md)를 따른다.

- `17`의 SceneID는 `FStorySceneID` 후보로 사용한다.
- `17`의 `Asset` 값은 StorySceneAsset 분리 후보로 사용한다.
- `17`의 `다음` 값은 `NextSceneID` 또는 `@goto` 전환 후보로 사용한다.
- D-Day 계열 SceneID는 `13_D-Day_스크립트_분할.md`의 `DDay_00Entry`, `DDay_04Gate`, `DDay_05True` 같은 기존 ID를 재사용하고 새 이름을 만들지 않는다.
- 진엔딩 후일담 계열 SceneID는 `TE_00_TimeSkip`, `TE_01_Wake`, `TE_10_Credit`처럼 `TE_` 접두어를 유지한다.
- 제작 문서에서 임시 설명이 필요하더라도 StoryState 키는 `HayeonTrust`, `IsSohaResolved`, `HasSeorinClue`, `HasReachedTrueEndingMonday`처럼 짧은 PascalCase로 고정한다.

### 4.1 날짜 / 회차 상태

필요 상태:

| 상태 | 예시 | 설명 |
| --- | --- | --- |
| 현재 날짜 | `D-25`, `D-14`, `D-1`, `D-Day`, `D+3` | 이벤트 개방과 루프 시작점, 축제 1일차/2일차, 방학식 예정일 표시에 사용 |
| 현재 주차 | `Week1`, `Week2`, `Week3`, `FinalWeek` | 이벤트 단계 관리용 |
| 첫 회차 여부 | `IsFirstRun` | 첫 회차 전용 1주 프롤로그 표시 여부 |
| 루프 횟수 | `LoopCount` | 반복 연출, 히든 엔딩 조건에 사용 |
| D-Day 진입 여부 | `DDayReached` | 엔딩 분기 트리거 |
| 방학식 예정 인지 여부 | `CeremonyKnown` | D+3 월요일 방학식 안내와 미루 멘토링 종료 불안에 사용 |

요구 기능:

- 날짜 진행
- 특정 날짜 범위 이벤트 개방
- 첫 회차와 루프 이후 시작점 분리
- D-1 도달 시 축제 1일차 부스 운영 이벤트와 D-Day 진입 전 최종 정산 처리
- D-Day 도달 시 축제 2일차 선택과 엔딩 조건 평가
- D+3 월요일 방학식은 예정일로 관리하되, 실패 루프에서는 실제 일반 진행 날짜로 진입하지 않음

### 4.2 캐릭터 진행도

필요 캐릭터:

- 은하연
- 한소하
- 백서린
- 나미루

각 캐릭터별 상태:

| 상태 | 설명 |
| --- | --- |
| 관계도 / 호감도 | 하연과의 상호 마음 확인과 일반 히로인 이벤트 진행에 사용 |
| 이벤트 단계 | Week별 이벤트 진행도 |
| 클라이맥스 해결 여부 | 진엔딩 조건 B에 사용 |
| 관계 균열 단계 | 미련, 회피, 의존, 기억 혼선 연출 강도에 사용 |
| 마지막 선택 날짜 | 병렬 이벤트 선택 균형 확인용 |

주의:

- 일반 히로인의 관계도는 연애 성공을 뜻하지 않는다.
- 일반 히로인 진행도는 `기존 관계 정리`와 `미련/회피/의존 해소`를 의미한다.

| `HasReachedTrueEndingMonday` | 진엔딩에서 D+3 월요일에 실제 도달 |
| `CanEnterTrueEndingCredit` | 진엔딩 크레딧 진입 가능 |
| `HasSeenCeremony` | 진엔딩 이후 D+3 월요일 방학식 참석 |
| `CanEnterDDayTrueEnding` | 하연과의 상호 마음 확인, D-1 하연 부스 운영, D-2 말하지 못한 예감, 기존 관계 정리, 사고 단서 조건을 충족해 D-Day 진엔딩 선택지 개방 |

요구 기능:

- 플래그 설정 / 해제 / 조회
- Branch에서 플래그 기반 분기
- 여러 플래그 조합으로 선택지 개방

D-1 밤 정산과 D-Day 낮 보정의 구체 점수·우선순위 기준은 [D-1/D-Day 조건 수치화](../스토리/12_D-1_D-Day_조건_수치화.md)를 따른다. 해당 문서는 `HayeonTrust`, `HayeonPace`, 기존 관계 `Is*Resolved` 플래그, 사고 단서 수, `Avoid`를 조합해 폐장 후 최종 분기를 평가하는 기준표다.

## 5. 핵심 시스템 요구사항

### 5.1 날짜 기반 이벤트 허브

미연시의 기본 루프는 날짜별 허브다.

예시 선택지:

```text
[하연과 시간을 보낸다]
[소하와 등교한다]
[서린을 돕는다]
[미루의 멘토링을 봐준다]
[축제 준비를 조사한다]
```

필요 기능:

- 현재 날짜와 캐릭터 진행도에 따라 선택 가능한 이벤트 목록 생성
- 이미 완료한 이벤트 제외 또는 압축판 제공
- 특정 이벤트 우선순위 처리
- 선택 결과로 StoryFlow Scene 시작

StoryFlow 연결:

- 날짜 허브는 별도 Scene으로 구성 가능
- 각 선택지는 Transition으로 이벤트 Scene에 진입
- 이벤트 종료 후 다시 허브 Scene으로 복귀

### 5.2 조건 기반 선택지

D-Day와 주요 클라이맥스에서는 조건에 따라 선택지가 달라진다.

예:

조건 부족:

```text
[아무 일도 없었던 것으로 한다]
[선택하지 않는다]
[여기에 남는다]
```

조건 충족:

```text
[하연에게 돌아간다]
```

완전 조건 충족:

```text
[현실의 하연에게 돌아간다]
```

진엔딩 복귀 처리:

- 금요일 밤 D-Day 선택 이후 토요일 날짜 허브는 생성하지 않는다.
- `DDay_05True` 이후 `TE_00_TimeSkip`로 전환하고, 바로 일요일 오후 병실 Scene인 `TE_01_Wake`로 이어진다.
- 재윤은 큰 외상은 없지만 알 수 없는 이유로 깨어나지 못했던 상태로 처리한다.
- 일요일 병실 Scene 이후 `HasReachedTrueEndingMonday`를 켜고 D+3 월요일 방학식 에필로그 Scene을 개방한다.
- `CanEnterTrueEndingCredit`가 켜진 뒤 진엔딩 크레딧으로 연결한다.

필요 기능:

- 선택지별 표시 조건
- 선택지별 비활성 조건 또는 숨김 조건
- 선택 시 상태 변경
- 조건 미충족 시 대체 선택지 제공

StoryFlow 연결:

- BranchTemplate에서 상태를 조회해 출력 인덱스를 선택할 수 있어야 함
- 또는 선택지 UI가 상태 조건을 평가하고, 결과를 StoryFlow에 전달해야 함

### 5.3 병렬 진행도 관리

한 명씩 공략하는 구조가 아니므로 병렬 상태가 필요하다.

필요 기능:

- 캐릭터별 이벤트 단계 저장
- 선택하지 않은 캐릭터의 관계 균열 단계 증가
- 특정 주차 이후 클라이맥스 이벤트 개방
- 하연과의 상호 마음 확인, D-1 하연 부스 운영 이벤트와 여러 캐릭터 조건이 D-Day 진엔딩 조건에 합산

권장 모델:

```text
CharacterProgress
- CharacterID
- AffectionOrBond
- EventStage
- DistortionLevel
- ClimaxDone
- LastEventDay
```

### 5.4 루프 처리

루프는 메인 반복 플레이를 강요하는 시스템이 아니라, 실패 후 재도전 장치다.

필요 기능:

- D-Day 실패 시 D-25로 복귀
- D+3 월요일 방학식은 실패 루프에서 도달하지 못하는 예정일로 유지
- 첫 회차 전용 프롤로그는 반복하지 않음
- 기억 조각 / 엔딩 기록 / 일부 플래그 유지
- 일반 관계도와 날짜는 루프 시작점에 맞게 재초기화

유지할 상태:

- 엔딩 기록
- 기억 조각
- 히든 엔딩 여부
- 기억 혼선 / 루프 피로
- 일부 메타 플래그

초기화할 상태:

- 현재 날짜
- 일반 이벤트 진행도
- 일시적 선택 플래그
- D-Day 진입 상태

### 5.5 이벤트 재방문 압축

루프 피로도를 줄이기 위해 필요하다.

필요 기능:

- 이벤트 최초 관람 여부 저장
- 재관람 시 압축 이벤트로 전환
- 플레이어가 스킵/요약을 선택 가능
- 단, 중요한 선택지는 다시 제공

예:

- 최초 관람: 전체 대화와 연출
- 2회차: 핵심 문장 + 선택지만 표시
- 기억 조각 보유 시: 바로 분기 선택지 제공

### 5.6 기억 혼선과 루프 피로

## 9. VisualNovelPlugin 설계 방향

현재 StoryFlow는 Scene/Shot/Branch 실행에 강점이 있으므로, 미연시 시스템은 별도 `VisualNovelPlugin`으로 둔다. StoryFlow 코어를 미연시 전용으로 바꾸기보다, `VisualNovelPlugin`이 `StoryState`와 VN용 Shot/Branch를 제공하고 StoryFlow에 연결한다.

`VisualNovelPlugin` 1차 후보:

1. `UVNStateSubsystem`
   - 날짜, 슬롯, 플래그, 기억 조각, 캐릭터 진행도, 엔딩 기록 관리
2. `UVNEventHubSubsystem`
   - 날짜/슬롯/조건별 이벤트 목록 구성과 시작 SceneID 반환
3. `UVNConditionEvaluator`
   - 조건 기반 선택지와 Branch 평가
4. `UVNLoopManager`
   - 루프 초기화, 유지/초기화할 상태 구분
5. `UVNEndingManager`
   - 엔딩 조건 평가와 엔딩 기록
6. `UVNSaveBridge`
   - `FStoryFlowRef`와 `FVNStoryState` 저장/복원 연결

VN용 StoryFlow 확장 후보:

| 확장 | 목적 | 구현 위치 |
| --- | --- | --- |
| `UVNDialogueShot` | 대사/캐릭터/배경/음악/SFX 재생 | `VisualNovelPlugin` |
| `UVNChoiceShot` | 플레이어 선택지 표시와 선택 결과 저장 | `VisualNovelPlugin` |
| `UVNConditionBranch` | `StoryState` 조건을 읽고 출력 인덱스 선택 | `VisualNovelPlugin` |
| `UVNFlagShot` | 특정 Shot 완료 시 플래그/기억 조각 갱신 | `VisualNovelPlugin` |
| `UVNEventHubSubsystem` | 날짜/슬롯/조건에 맞는 이벤트 목록 구성 | `VisualNovelPlugin` |

최소 결론:

- StoryFlow의 큰 틀은 유지한다.
- 미연시용 상태/허브/선택지/조건 평가는 `VisualNovelPlugin`으로 추가한다.
- StoryFlow 코어 변경은 저장 브리지나 편의 API가 부족할 때만 작게 진행한다.

## 10. 다음 설계 과제

`VisualNovelPlugin` 1차 최소 범위, 상태/이벤트 데이터, `UVNDialogueShot` / `UVNChoiceShot` 입력 필드, D-Day 최소 검증 시나리오는 확정되었다. 다음 과제는 구현 착수와 에디터 검증 규칙이다.

2. 데이터 검증 규칙을 에디터 Validator 작업 목록으로 분해
